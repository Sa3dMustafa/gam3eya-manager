# مدير الجمعيات (Gam3eya Manager)

نظام كامل لإدارة الجمعيات المالية الدورية (ROSCA) — مستخدم واحد، بدون تسجيل دخول.

واجهة عربية بالكامل (RTL)، مبنية بـ **Next.js 15 (App Router) + TypeScript + Prisma 7 + PostgreSQL + Tailwind CSS + Zustand + Recharts**.

يمكن تشغيل التطبيق محليًا على جهازك، أو رفعه على الإنترنت عبر **Vercel** (مدعوم بالكامل).

---

## 1. المتطلبات

- **Node.js 20 أو أحدث** — تحقق بكتابة: `node -v`
- **npm** (يأتي مع Node.js)
- **قاعدة بيانات PostgreSQL** — محلية (عبر Docker أو تثبيت مباشر) أو سحابية مجانية (Vercel Postgres، Neon، Supabase، إلخ)
- اتصال بالإنترنت أثناء التثبيت (لتحميل الحزم ومحرك Prisma)

---

## 2. التشغيل محليًا

### أ) تشغيل قاعدة بيانات Postgres محليًا (أسهل طريقة: Docker)

```bash
docker run --name gam3eya-db -e POSTGRES_PASSWORD=password -e POSTGRES_DB=gam3eya -p 5432:5432 -d postgres:16
```

أو استخدم قاعدة بيانات سحابية مجانية (مثل [Neon](https://neon.tech) أو [Vercel Postgres](https://vercel.com/storage/postgres)) واحصل على رابط الاتصال (connection string) منها.

### ب) إعداد رابط الاتصال

افتح ملف `.env` في جذر المشروع وضع رابط قاعدة البيانات:

```env
DATABASE_URL="postgresql://postgres:password@localhost:5432/gam3eya"
```

### ج) التثبيت والتشغيل

```bash
# 1. الدخول لمجلد المشروع
cd gam3eya-manager

# 2. تثبيت كل الحزم (سيقوم تلقائيًا بتوليد Prisma Client عبر postinstall)
npm install

# 3. إنشاء الجداول في قاعدة البيانات بناءً على المخطط (Schema)
npm run db:push

# 4. (اختياري لكن يُنصح به) إضافة بيانات تجريبية للتعرف على التطبيق
npm run db:seed

# 5. تشغيل التطبيق في وضع التطوير
npm run dev
```

افتح المتصفح على: **http://localhost:3000**

> إذا واجهت رسالة خطأ من Prisma أثناء `npm install` بخصوص تحميل ملفات المحرك (engine)، تأكد أن جهازك متصل بالإنترنت وأن الجدار الناري لا يحجب `binaries.prisma.sh`، ثم أعد المحاولة بـ `npx prisma generate`.

### مشكلة شائعة على Windows: أخطاء `ENOENT ... _buildManifest.js.tmp` عند `npm run dev`

هذه مشكلة معروفة في Turbopack على Windows غير متعلقة بكود المشروع، وسببها الغالب تضارب مع برنامج حماية (Antivirus) أو OneDrive يقوم بقفل/فحص الملفات أثناء كتابتها في مجلد `.next`. للحل:

```bash
# أوقف الخادم (Ctrl+C) ثم نفّذ:
rmdir /s /q .next   # في PowerShell: Remove-Item -Recurse -Force .next
npm run dev
```

إذا تكررت المشكلة:
- استثن مجلد المشروع بالكامل من فحص برنامج الحماية (Windows Defender أو غيره).
- تجنّب وضع المشروع داخل مجلد متزامن مع OneDrive/Google Drive.
- كحل أخير، شغّل بدون Turbopack: `npx next dev` (بدون `--turbopack`).

## 3. الرفع على الإنترنت (Vercel)

### الخطوة 1: رفع الكود إلى GitHub

ارفع مجلد المشروع كمستودع (repository) على GitHub (أو GitLab/Bitbucket).

### الخطوة 2: إنشاء قاعدة بيانات Vercel Postgres

1. من [لوحة تحكم Vercel](https://vercel.com/dashboard)، روح لتبويب **Storage**.
2. اختر **Create Database** ← **Postgres**.
3. اختر اسمًا والمنطقة (Region) الأقرب لك، ثم أنشئها.
4. بعد الإنشاء، اربطها بمشروعك (Connect to Project) — هذا يحقن متغيرات البيئة (`POSTGRES_PRISMA_URL`, `POSTGRES_URL_NON_POOLING`, إلخ) تلقائيًا.

### الخطوة 3: ضبط متغيرات البيئة في Vercel

في إعدادات المشروع على Vercel (Settings → Environment Variables)، أضف:

| المتغير | القيمة |
|---|---|
| `DATABASE_URL` | نفس قيمة `POSTGRES_PRISMA_URL` التي أنشأها Vercel (الرابط المُجمَّع/pooled) |
| `DIRECT_URL` | نفس قيمة `POSTGRES_URL_NON_POOLING` التي أنشأها Vercel (رابط مباشر، لأوامر الـ migration) |

> لو ربطت قاعدة البيانات من تبويب Storage مباشرة، Vercel أحيانًا يحقن `POSTGRES_PRISMA_URL` و `POSTGRES_URL_NON_POOLING` تلقائيًا — في هذه الحالة فعليًا التطبيق سيستخدمها تلقائيًا (راجع `src/lib/prisma.ts`)، لكن يُفضّل أيضًا إضافة `DIRECT_URL` بنفس قيمة `POSTGRES_URL_NON_POOLING` لضمان عمل أوامر الـ migration بسلاسة من جهازك المحلي مستقبلًا.

### الخطوة 4: استورد المشروع على Vercel

1. من [لوحة تحكم Vercel](https://vercel.com/new)، استورد المستودع من GitHub.
2. Vercel هيكتشف أنه مشروع Next.js تلقائيًا — لا حاجة لأي ضبط إضافي.
3. اضغط **Deploy**.

### الخطوة 5: إنشاء الجداول في قاعدة البيانات السحابية

بعد أول Deploy، شغّل من جهازك (بعد ضبط `DATABASE_URL`/`DIRECT_URL` في `.env` المحلي بنفس قيم Vercel):

```bash
npm run db:push
npm run db:seed   # اختياري
```

أو استخدم [Vercel CLI](https://vercel.com/docs/cli) لتنفيذ الأمر مباشرة في بيئة الإنتاج:

```bash
vercel env pull .env.production.local
npx dotenv -e .env.production.local -- npx prisma db push
```

بعد ذلك، التطبيق هيكون متاح بالكامل على الرابط اللي يديك Vercel، وكل البيانات (الجمعيات، الأعضاء، الدفعات، **بما فيها مرفقات الإيصالات/الصور**) محفوظة بأمان في قاعدة البيانات السحابية.

## 5. أين تُخزَّن بياناتي؟

- **كل البيانات** — الجمعيات، الأعضاء، الأدوار، الدفعات، الملاحظات، **ومرفقات الإيصالات/الصور (محفوظة كـ base64 داخل قاعدة البيانات نفسها)** — كل ده في قاعدة بيانات PostgreSQL واحدة (محلية أو سحابية حسب اختيارك).
- لا تخزين ملفات على القرص بعد الآن، لذلك التطبيق متوافق بالكامل مع بيئات السيرفرليس (Serverless) مثل Vercel.

للنقل أو الاحتفاظ بنسخة احتياطية: استخدم زر **"تنزيل نسخة احتياطية"** من صفحة الإعدادات (يُصدّر **كل** البيانات، بما فيها المرفقات، في ملف JSON واحد)، ثم **"استرجاع من ملف"** لاستعادتها على أي نسخة من التطبيق متصلة بأي قاعدة بيانات.

```
src/
  app/
    page.tsx                  # لوحة التحكم الرئيسية
    gam3eyat/                 # قائمة وتفاصيل الجمعيات
    members/[id]/             # صفحة العضو الكاملة

    archive/                  # الجمعيات المؤرشفة
    settings/                 # المظهر + النسخ الاحتياطي

  components/

    dashboard/ gam3eya/ member/ payment/ round/ shared/
  lib/
    prisma.ts                 # عميل Prisma + إعداد Driver Adapter (pg) + تجميع الاتصالات لـ Vercel
    calculations.ts           # كل المنطق المالي (التحصيل، التأخر، الإحصائيات الذكية)
    round-sync.ts             # مزامنة الأدوار تلقائيًا مع ترتيب استلام الأعضاء
    validations.ts            # مخططات Zod للتحقق من المدخلات
    format.ts                 # تنسيق العملة والتواريخ بالعربية

prisma.config.ts                # تهيئة Prisma 7 (رابط قاعدة البيانات، مسار الترحيلات، أمر البذر)
```

## 7. ملاحظة أمنية بخصوص حزمة xlsx

تستخدم ميزة "تصدير Excel" حزمة `xlsx` (SheetJS) كما هو محدد في متطلبات المشروع. توجد في هذه الحزمة ثغرات معروفة (إمكانية ReDoS وPrototype Pollution) لا يوجد لها إصلاح رسمي حتى الآن من المُطوّر. بما أن هذا التطبيق لمستخدم واحد وبدون رفع ملفات خارجية لمحرك xlsx، فإن المخاطر العملية منخفضة جدًا. إن كانت لديك حساسية خاصة تجاه هذا، يمكنك استبدال الحزمة بأخرى مثل `exceljs` لاحقًا.

## 8. حدود مهمة عند التشغيل على Vercel

- **حجم الطلبات والاستجابات:** تفرض Vercel حدًا أقصى 4.5 ميجابايت لحجم أي طلب أو استجابة لدالة سحابية واحدة. بسبب هذا:
  - الحد الأقصى لحجم مرفق الدفعة الواحد (إيصال/صورة) هو **3 ميجابايت** (مضبوط في `src/app/api/upload/route.ts`).
  - تنزيل النسخة الاحتياطية يُرسل كـ stream لتفادي هذا الحد، لكن **استرجاع** نسخة احتياطية كبيرة جدًا (نادر الحدوث) قد يفشل إذا تجاوز حجم الملف هذا الحد، لأن طلب الاستعادة (POST) يُرسل دفعة واحدة.
  - يمكنك دائمًا اختيار استثناء المرفقات من النسخة الاحتياطية (مفتاح في صفحة الإعدادات) لتصغير حجم الملف.
- **مهلة تنفيذ الدالة:** العمليات الثقيلة (مثل تصدير Excel لجمعية ضخمة جدًا بمئات الأعضاء والأدوار) من المتوقع أن تكتمل في أقل من ثانية إلى ثوانٍ قليلة، وهذا أقل بكثير من الحد الافتراضي لمهلة Vercel.
